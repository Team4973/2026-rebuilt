## Project Overview

This is a RobotPy (Python WPILib) FRC robot project implementing a swerve drivetrain using CTRE Phoenix 6 hardware (TalonFX motors, CANcoders, Pigeon 2 IMU). The project uses the WPILib command-based framework.

## Build and Run Commands

This project uses `uv` for Python dependency management.

```bash
# Sync dependencies
uv sync

# Run robot simulation
uv run python -m robotpy sim

# Run tests
uv run python -m robotpy test

# Run a single test
uv run pytest tests/pyfrc_test.py

# Deploy to robot
uv run python -m robotpy deploy
```

## Architecture

### Entry Point
- `robot.py` - `MyRobot` extends `TimedCommandRobot`. Instantiates `RobotContainer`, runs the CommandScheduler each periodic cycle, and manages autonomous command lifecycle. Also sets up `HootAutoReplay` for timestamp/joystick log replay.

### Core Component
- `robotcontainer.py` - All subsystems, swerve requests, controller bindings, and autonomous logic live here.
  - `configureButtonBindings()` - Full drivetrain + launcher bindings (currently commented out in `__init__`)
  - `configureDpadBindings()` - Active climber bindings: D-pad up/down → climber at ±0.5 speed via `whileTrue`/`onFalse` Trigger API
  - `getAutonomousCommand()` - Seeds field-centric heading, drives forward 0.5 m/s for 5 seconds, then idles

### Subsystems (`subsystems/`)
- `command_swerve_drivetrain.py` - `CommandSwerveDrivetrain` wraps Phoenix 6's `SwerveDrivetrain` as a WPILib Subsystem. Handles operator perspective (alliance-aware), SysId routines, and simulation fast update loop.
- `climber.py` - `Climber` subsystem. Single TalonFX (CAN ID 34) using `DutyCycleOut` control. Methods: `set_speed(float)`, `stop()`.
- `launcher.py` - `Launcher` subsystem. Single TalonFX (CAN ID 47, default) using `DutyCycleOut` control. Currently commented out in `RobotContainer`. Same interface as `Climber`.

### Generated Code
- `generated/tuner_constants.py` - Do not hand-edit. Generated by CTRE Tuner X Swerve Project Generator. Contains `TunerConstants` (PID gains, gear ratios, CAN IDs, module positions, encoder offsets), `TunerSwerveDrivetrain`, and `create_drivetrain()`.

### Telemetry (`telemetry/`)
- `telemetry/telemetry.py` - `Telemetry` class. Publishes swerve drive state (pose, speeds, module states/targets/positions) to both NetworkTables (`DriveState/` table) and SignalLogger. Also publishes a `Field2d` pose and per-module `Mechanism2d` visualizations to SmartDashboard.
- `telemetry/powerdistribution/PowerDistributionHelper.py` - Module-level script that reads PDH (REV, CAN ID 1) values: voltage, total current, faults, version, and per-channel currents (1–24). Imported as a module by `PowerDistributionNT`.
- `telemetry/powerdistribution/PowerDistributionNT.py` - `PowerDistributionNT` publishes PDH values to NetworkTables topics via `publishValues()`. Not currently wired into `RobotContainer`.

## Hardware CAN IDs
- Pigeon 2 IMU: CAN ID 0
- Climber TalonFX: CAN ID 34
- Launcher TalonFX: CAN ID 47 (default; currently unused)
- Power Distribution Hub: CAN ID 1 (REV)
- Swerve module drive/steer motors and CANcoders: defined in `generated/tuner_constants.py`

## Key Dependencies
- `robotpy[commands2,sim]~=2026.1` - WPILib and command framework
- `phoenix6~=26.1` - CTRE Phoenix 6 motor controller library

## Notes
- The `telemetry` import in `robotcontainer.py` is `from telemetry import telemetry` (package → module), not a top-level import — keep the `telemetry/` directory structure intact.
- Logs are written to `./logs/` on the RoboRIO via Phoenix hoot logging.
- `simgui-ds.json` configures the simulation driver station layout.
